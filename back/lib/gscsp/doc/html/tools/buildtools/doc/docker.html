4.1.4. Docker functionality¶

Buildtools adds support for building GomSpace assets and projects within docker containers.

4.1.4.1. Prerequisites¶

The user is required to have docker installed. On Ubuntu 16.04 this can be achieved by:

sudo apt-get install docker.io

The user is required to be in the docker group: (This step requires that you logout and back in)

sudo adduser <yourname> docker

To use the internal Gomspace Docker registry the certificate must be downloaded:

wget http://jw-file-srv/gs-docker-reg-server.crt
sudo mkdir -p /etc/docker/certs.d/gs-docker-reg:443
sudo mv gs-docker-reg-server.crt /etc/docker/certs.d/gs-docker-reg:443/ca.crt
sudo systemctl restart docker

4.1.4.2. Using Docker with Buildtools¶

Buildtools uses docker by default.

When calling gsbuildtools_bootstrap.py the usual waf symlink is replaced with a symlink to buildtools/bin/dock_waf, the link is still called waf. This script will prepend all calls to waf with the correct docker command and arguments.

As gsbuildall.py also uses the symlinked waf commands executed by gsbuildall.py will also be executed inside docker containers.

4.1.4.2.1. Mounted Environment¶

By default buildtools will mount the current project directory and the path to the bootstrapped buildtools. If you want to use e.g. symlinks between different projects you can tell buildtools to mount a different path. This is handled by the internal function:

gs.buildtools.docker_functions._get_workspace_root(path='.')¶

Search for a file called .gs_workspace_root recursively up from current dir

If you have a common folder holding all your gomspace repositories called ‘workspace’, you could create ‘workspace/.gs_workspace_root’ and buildtools will then mount that instead of the current project and the buildtools path.

Returns:dirname if found otherwise None

4.1.4.3. Programmer Support¶

Please refer to the internal function in buildtools:

gs.buildtools.docker_functions._get_device_args()¶

If any known programmers are present add them as a resource for docker and add the user to the groups required (plugdev and dialout). Please note that when a PEAK dongle is seen all network interfaces are added to the container.

Currently supported programmers and tools are:

AVR Dragon

Aardvark dongle

Olimex JTAG programmer

PEAK System PCAN-USB

4.1.4.4. Binding to Versions¶

Binding to a specific version of a docker image is handled by the internal function:

gs.buildtools.docker_functions._resolve_required_version(image, fn='dependencies.json')¶

resolve the version required by the project If no versions can be extracted from the project the versions from buildtools will be used Add something like the following as entry to dependencies.json

“type”: “docker”,

“min_version”: “x.y.z”,

“name”: “container_name”

The same version will be used for all pulled images

4.1.4.5. Controlling Container Versions¶

Each version of buildtools has a binding towards a set of container versions. These versions are configured in dependencies.json of buildtools. Each project/assert can override these versions by specifying their own version in dependencies.json.

In the dependencies.json example below, the container pasw-linux is bound to version 0.7.0.

Note

Use the special type “docker”

{
    "dependencies": [
        {
            "min_version": "0.7.0",
            "name": "pasw-linux",
            "type": "docker"
        }
    ]
}

4.1.4.6. Building without Docker¶

It is possible to build locally by bootstrapping the project with the --no-docker argument.

./tools/buildtools/gsbuildtools_bootstrap.py --no-docker

When --no-docker is used the standard waf is used and build commands are done directly on the local on the local machine.
